# synthonyx-kit-storage

Backend-agnostic storage traits for the Synthonyx Kit. Phase 1 defines the
trait surface only; concrete backends ship as separate crates in Phase 2
(`synthonyx-kit-storage-memory`, `-postgres`, `-sqlite`, `-redis`).

## Contents

- `StorageKey`, `Codec`, `CodecError`, `StorageError`, `KeyId`.
- `Backend` — async raw byte-level KV. `Send`-bounded `BackendSend` variant
  generated by `trait_variant`.
- `TransactionalBackend` — marker; concrete API in Phase 2.
- `EncryptedBackend` — `encrypts_at_rest()`, `key_id()` for rotation
  tracking.
- `RetentionBackend` — `apply_retention(&RetentionPolicy)`.
- `ErasureBackend` — `erase_subject(...) -> ErasureReport` (GDPR Art. 17).
- `StorageValue`, `StorageMap`, `StorageDoubleMap` — typed storage shape
  RTMs declare via associated types on their `Config` extension trait.

The compliance hooks (`EncryptedBackend`, `RetentionBackend`,
`ErasureBackend`) are nailed down at the trait level in Phase 1 so RTMs
authored against the kernel today will run unchanged against Phase 2
backends.
